# Platform Port Model — Implementation Tracker

**Status: planning tracker — implementation not started.** Companion to the
design note [`platform-port-model.md`](platform-port-model.md), which is the
authority for *what* and *why*; this document is the *how* and *in what order*
(ai-instructions §15). Phases are sequenced additive-first → rewire → delete so
every phase is independently shippable, build-green, and behavior-preserving
until the new path is proven (CLAUDE.md Regression Prevention). Mark `[x]` per
task as completed — do not batch. CLAUDE.md's doc index is not updated until the
change lands.

## Architecture-First Note (ai-instructions §18)

The target is the design note's model — an explicit, generated `Ports` table as
the single `name → nic_index` mapping. The existing `vm_interface_map` stride
code reflects the *old* model; it is deleted, not bent to fit. Design each phase
from the note as if the stride code did not exist, then reuse what survives.

## Resolved Concerns (ai-instructions §14 — resolved, not deferred)

| # | Concern | Resolution |
|---|---------|-----------|
| 1 | `vm_interface_map`'s fate | **Removed** (DPN §40). `nic_index` is the stored mapping; the generator assigns it by `port_config.ini` row order. The stride-4 divisibility *validation* is subsumed — the table lists only ports that exist. |
| 2 | VPP **synthesizes** `port_config.ini` from `PortStride` (RCA-013), via `patches/vpp/always/{port_config.ini,port_entries,sonic_vpp_ifmap.ini}.tmpl` | Rewrite the three templates to range over the first `dataNICs` rows of the platform's `Ports` table (name + nic_index) instead of computing from a stride. RCA-013's invariant holds: the table is the per-variant deployment artifact (the VPP variant's table lists its post-rename `Ethernet0,1,2…`). |
| 3 | `parseLinuxEthIndex` has a **second use** (link.go:101,110, coalesced-host path) | That use maps a coalesced host's `ethN` to a NIC base — unrelated to platform ports. **Keep `parseLinuxEthIndex`**; only `parseEthernetIndex` (stride math) is deleted. |
| 4 | Host platforms have **variable NICs** — does an explicit table fit? | Yes. A lab host has a *defined* NIC budget, not unbounded. `alpine-host`/`vjunos-router` get **hand-authored** tables up to their budget — same as switches having a fixed complement. Non-SONiC tables are authored, not generated (no `port_config.ini`). |
| 5 | Schema is **newtcon-consumed** (`schema_registry.go:243`, read-only) — field change is UI-visible | Reflection-derived from `PlatformSpec{}`, so `ports[]` appears and `vm_interface_map` disappears automatically. Phase 0 newtcon heads-up; no newtcon code change (read-only view). Update `schema_meta_test.go`. |
| 6 | Platform CRUD over the wire | **None exists** — write endpoints removed (handler.go:119); platforms are on-disk, generated by `newtron platform generate`. No write-handler work. |
| 7 | Transient state where both `vm_interface_map` and `ports[]` exist (Phases 1–2) | **Not** a DPN §40 dual-API — a staged migration across consecutive PRs; the old field is deleted in Phase 3, never released as a permanent alternative. |

## Deferred (ai-instructions §14.3 — out of scope, with reasons)

- **vJunos generator** — depends on the unresolved newtron↔netconf.pl integration
  decision (note §10 Q1). The schema supports arbitrary names from Phase 1, so
  vJunos works once a table is authored; the generator waits. In-tree
  `vjunos-router.json` gets a hand-authored table in Phase 1 (its topologies have
  `links:[]` today, so nothing resolves against it yet).
- **No-gaps (RCA-020) relaxation** — the explicit table enables it (contiguity
  becomes checkable), but relaxing is a separate decision (note §10 Q4).
- **Breakout logical sub-ports** — the table enumerates base front-panel ports;
  `Breakouts []string` stays a separate dimension (note §10 Q3).

> **Deviation from the original plan:** host-platform tables (`alpine-host`,
> `vjunos-router`) were moved to **Phase 2**. Rationale: the SONiC switch tables
> are pure projections of a known stride convention (zero behavioral decision),
> but the host "linux"-scheme → bounded-table conversion is a *semantic* choice
> (port budget; the `eth0`-in-links subtlety) entangled with Phase 2's
> validation behavior and cold-deploy gate. It belongs with the consumer change
> that exercises it, not the additive storage phase. Observed today: host eth
> usage maxes at `eth2`; no topology uses `ge-/xe-` naming (vJunos wires
> `eth1/eth2` via the linux scheme).

## Sequencing

Each phase is one PR. Phases 1 and 3 are loopback-fast; **Phase 2 carries the
cold-deploy gate** and must be validated on a freshly-deployed topology before
Phase 3 deletes the old path. Do not merge Phase 3 until Phase 2's cold-deploy
validation is green — the old field is the fallback reference until then.
