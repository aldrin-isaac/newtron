 Future Iteration: Remote Client Support                                                                                                                                        │
│                                                                                                                                                                                │
│ This iteration assumes client and server share the same filesystem (spec directory                                                                                             │
│ is a local path). The next iteration must support remote clients where:                                                                                                        │
│                                                                                                                                                                                │
│ 1. Specs registered via API: Network registration accepts either a local path                                                                                                  │
│ (same machine) or specs as inline JSON (remote client). POST /network body:                                                                                                    │
│ {"id": "lab", "spec_dir": "/path/to/specs"}          // local                                                                                                                  │
│ {"id": "lab", "specs": {"services": {...}, ...}}      // remote (inline)                                                                                                       │
│ 2. Spec authoring via API: CreateService, CreateIPVPN, etc. already work via                                                                                                   │
│ HTTP. For remote clients, these become the only way to populate specs (no                                                                                                      │
│ shared filesystem to read from).                                                                                                                                               │
│ 3. Model validation with constraints: Spec types (ServiceSpec, IPVPNSpec, etc.)                                                                                                │
│ need explicit validation rules — required fields, value ranges, cross-references                                                                                               │
│ (e.g., a service's ipvpn must exist). Today validation is minimal (checked at                                                                                                  │
│ apply time). For remote clients, validation must happen at spec creation time                                                                                                  │
│ with clear error messages. This means:                                                                                                                                         │
│   - Each spec type gets a Validate() error method                                                                                                                              │
│   - Cross-reference validation: "ipvpn 'foo' referenced by service 'bar' does not exist"                                                                                       │
│   - Value constraints: VNI range 1-16777215, VLAN range 1-4094, ASN range, IP format                                                                                           │
│   - The API returns 400 with structured validation errors
